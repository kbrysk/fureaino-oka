# CMS 3者比較：microCMS / Sanity / Keystatic

**前提**
- 1年後に事業売却（バイアウト）の可能性を視野に入れる
- 自前でコンテンツを作成し、地道に投入していく運用（Gemini 等でリサーチ・執筆 → CMS に追加）
- フクロウの相槌・LINE誘導（文頭・文末）は現状の設計のまま残す

**比較対象**
| CMS | 種別 | 契約・コスト |
|-----|------|--------------|
| **microCMS** | 外部 SaaS（Headless CMS） | Hobby 無料 / Team 約4,900円/月〜 |
| **Sanity** | 外部 SaaS（Headless CMS） | Free 無料（10k doc・100GB帯域） |
| **Keystatic** | Git 直結型（リポジトリ内ファイル） | 契約不要（Vercel の範囲内で動作） |

---

## 1. オペレーション（運用）の効率性

### 1.1 日常の執筆フロー（1本ずつ追加）

| 観点 | microCMS | Sanity | Keystatic |
|------|----------|--------|-----------|
| **執筆の入口** | 管理画面（ブラウザ）。API キー・サービス URL を設定したサイトからアクセス | Sanity Studio（ブラウザ）。自前デプロイ or sanity.io ホスト | Next に埋め込んだ /keystatic（ブラウザ）。localhost または本番 URL |
| **編集体験** | ◎ 国産・日本語 UI。フォームが分かりやすく、非エンジニアでも操作しやすい | ◎ リッチなエディタ。Portable Text / ブロック編集。英語 UI が中心（カスタムで日本語化可） | ○ ブログ的な UI。Markdown / JSON をフォームで編集。英語 UI が中心 |
| **画像の扱い** | 管理画面からアップロード。CDN 配信。alt 等をフィールドで管理 | スタジオからアップロード。CDN＋画像最適化 API。alt 等をフィールドで管理 | リポジトリに保存 or 外部 URL。画像最適化は Next/ Vercel 側で対応 |
| **保存後の流れ** | 公開ボタンで即時反映。Next は ISR/on-demand で再取得 | 同様。公開で即時。Next は再取得で反映 | 保存 → Git にコミット（＋プッシュ）。Vercel がビルドして反映。数分かかる場合あり |
| **Cursor を開く必要** | 不要（管理画面のみで完結） | 不要（スタジオのみで完結） | 不要（/keystatic のみで完結）。※本番で使う場合は GitHub 連携設定が必要 |

→ **1本ずつ追加**では、**microCMS が日本語・運用の分かりやすさで有利**。Sanity は編集機能が強く、Keystatic は「保存＝コミット」の分だけ反映がビルド待ちになる。

### 1.2 一括投入（Gemini で大量作成 → 投入）

| 観点 | microCMS | Sanity | Keystatic |
|------|----------|--------|-----------|
| **API 一括投入** | ◎ REST API がシンプル。スクリプトで JSON を POST すればよい。ドキュメントが日本語 | ◎ GROQ / HTTP API。スクリプトで一括 POST 可能。英語ドキュメントが豊富 | △ API は「ファイルを読む」形。一括投入は「ローカルで .mdx/.json を生成 → git commit & push」するスクリプトが必要 |
| **100本・500本を一気に入れる** | ◎ スクリプトでループ POST。実行場所はローカル・GAS・CI などどこでも可 | ◎ 同様。ミューテーション API で一括投入が容易 | ○ スクリプトでファイル生成 → 一括コミット＆プッシュ。PR でまとめて入れる運用も可 |
| **失敗時のロールバック** | 管理画面 or API で該当記事を非公開・削除 | 同様。履歴・バージョンあり | Git の履歴で戻せる。差分が明確 |

→ **一括投入のしやすさ**は **microCMS ≒ Sanity ＞ Keystatic**。Keystatic は「ファイル＋Git」のため自動化は可能だが、API ほど直感的ではない。

### 1.3 スキーマ変更・メンテナンス

| 観点 | microCMS | Sanity | Keystatic |
|------|----------|--------|-----------|
| **フィールド追加** | 管理画面の API 設定から追加。既存記事は空 or デフォルトでよい | スキーマ（コード）を変更してデプロイ。既存データはマイグレーション不要のことが多い | keystatic.config のスキーマを変更。既存ファイルに新フィールドを足す場合は手動 or スクリプト |
| **フクロウの相槌（owlMessages）** | リスト型フィールドで追加すればよい | array of string で定義 | collection で array フィールドを定義 |
| **管理の煩雑性** | 契約・API キー・環境変数。サービスが1つ増える | 同様。プロジェクト ID・トークン・環境変数 | 契約なし。リポジトリと GitHub 連携（本番で /keystatic を使う場合）のみ |

→ **スキーマの柔軟性**は **Sanity ≒ Keystatic（コードで定義）＞ microCMS（管理画面で設定）**。**管理の煩雑さ（契約・アカウント数）**は **Keystatic が最小**。

### 1.4 運用効率のまとめ

| 項目 | microCMS | Sanity | Keystatic |
|------|----------|--------|-----------|
| 1本ずつ追加の手軽さ | ◎ 日本語・分かりやすい | ◎ 機能豊富 | ○ 問題なし。反映はビルド待ち |
| 一括投入のしやすさ | ◎ API が簡単 | ◎ API が強力 | ○ スクリプト＋Git で対応 |
| 契約・アカウントの数 | 1つ増える | 1つ増える | 増えない |
| 反映の速さ | 即時（ISR 等で取得し直し） | 即時 | コミット→ビルド後（数分） |

## 2. SEO プロフェッショナルとしての観点

### 2.1 初回表示・インデックス・Core Web Vitals

| 観点 | microCMS | Sanity | Keystatic |
|------|----------|--------|-----------|
| **コンテンツの取得タイミング** | ビルド時（SSG）or リクエスト時（ISR/on-demand）。Next が API を叩いて HTML に埋め込む | 同様。SSG/ISR で Sanity API を叩き、HTML に含める | ビルド時にリポジトリ内のファイルを読む（現状の getArticleBySlug と同様）。API 経由の遅延なし |
| **クローラーへの見え方** | ◎ サーバー側で HTML に含めるため、コンテンツはすべてクロール・評価対象 | ◎ 同様 | ◎ 同様。ファイル読みなのでレイテンシも小さい |
| **LCP・CLS** | 画像は microCMS CDN。Next の Image と組み合わせて最適化可能 | 画像は Sanity CDN＋画像 URL パラメータで最適化可能 | 画像は Next/ Vercel の最適化 or 自前配置。制御はすべて自前 |
| **TTFB・API レイテンシ** | 本番では ISR キャッシュで API を叩く頻度を抑えられる。転送量制限（Hobby 20GB/月）に注意 | 同様。Free で 100GB/月 と余裕あり | ファイル読みのため API レイテンシはなし。ビルド時間のみ |

→ **いずれも「サーバー側で HTML にコンテンツを含める」設計にできるため、SEO 上の表示・インデックスは同等**。差は **転送量制限（microCMS が厳しめ）** と **画像最適化の手間（Keystatic は自前でやる部分が多い）**。

### 2.2 メタ・構造化データ・URL

| 観点 | microCMS | Sanity | Keystatic |
|------|----------|--------|-----------|
| **title / description / OG** | API のフィールドをそのまま Next の metadata に渡せる。完全に制御可能 | 同様。GROQ で必要なフィールドだけ取得 | ファイル（frontmatter / JSON）のフィールドをそのまま使用。完全に制御可能 |
| **構造化データ（JSON-LD）** | 記事データを元に Next 側で自由に出力可能 | 同様 | 同様 |
| **URL・canonical・sitemap** | Next のルーティングで完全制御。CMS はデータの出所のみ | 同様 | 同様。ファイルベースなので generateStaticParams と相性が良い |
| **ロングテール・内部リンク** | 一覧 API で slug 一覧を取得し、内部リンク・サイトマップを生成可能 | 同様 | 現状と同様に index から slug 一覧を取得して生成可能 |

→ **SEO の制御性は 3 者とも同等**。タイトル・説明文・OG・構造化データ・URL はすべて Next 側で決められる。

### 2.3 コンテンツ更新・鮮度・E-E-A-T

| 観点 | microCMS | Sanity | Keystatic |
|------|----------|--------|-----------|
| **更新の手軽さ** | 管理画面で編集→公開で即反映（ISR なら次回取得で更新） | 同様 | 編集→コミット→プッシュ→ビルドで反映。即時ではない |
| **日付・更新日の管理** | フィールドで date / updatedAt を管理可能 | 同様。バージョン履歴もあり | 同様。frontmatter / JSON で管理 |
| **大規模なリライト** | API で一括更新可能 | 同様。ミューテーションで一括 | スクリプトでファイル一括置換 or 手動。Git で差分管理 |

→ **更新の「即時性」**は **microCMS ≒ Sanity ＞ Keystatic**。**鮮度シグナル・E-E-A-T のためのメタ制御**は 3 者とも問題なし。

### 2.4 SEO 観点のまとめ

| 項目 | microCMS | Sanity | Keystatic |
|------|----------|--------|-----------|
| 初回表示・インデックス | ◎ 問題なし | ◎ 問題なし | ◎ 問題なし（ファイル読みで遅延小） |
| メタ・構造化データ | ◎ 完全制御 | ◎ 完全制御 | ◎ 完全制御 |
| 転送量・スケール | △ Hobby 20GB/月。PV 増で要検討 | ◎ Free 100GB/月 で余裕 | ◎ 制限なし（Vercel の範囲内） |
| 更新の即時性 | ◎ 即時 | ◎ 即時 | ○ ビルド後（数分） |

→ **SEO プロとして見た場合、3 者とも「正しく実装すれば差はない」**。運用面では **転送量（microCMS に注意）** と **更新をどれだけ即時に出したいか（Keystatic は数分遅れ許容か）** が選択のポイント。

## 3. 将来的な事業の見通しに基づく最適な選択肢

### 3.1 事業売却（M&A）時の譲渡のしやすさ

| 観点 | microCMS | Sanity | Keystatic |
|------|----------|--------|-----------|
| **譲渡対象** | リポジトリ ＋ **microCMS アカウント**（API キー・契約の引き継ぎ） | リポジトリ ＋ **Sanity プロジェクト**（アカウント or プロジェクト移管） | **リポジトリのみ**。コード＋コンテンツがすべて Git 内 |
| **買い手の手続き** | アカウント受け取り・クレカ・請求先の変更。サービス継続の確認 | 同様。プロジェクト譲渡 or 組織移管 | リポジトリのオーナー変更のみ。**追加契約・移管作業がほぼなし** |
| **契約切れ・サービス終了リスク** | 理論上あり（極めてレア）。譲渡後は買い手が契約主体 | 同様 | **なし**。コンテンツはすべて自前資産 |
| **DD（デューデリ）での説明** | 「コンテンツは外部 SaaS にあり、アカウント譲渡で引き継ぎ」と説明が必要 | 同様 | 「コンテンツはすべてリポジトリ内にあり、譲渡＝リポジトリ移管で完結」と説明しやすい |

→ **譲渡のシンプルさ・資産の明確さ**は **Keystatic が最も有利**。**Sanity は海外標準で買い手の技術理解を得やすい**。**microCMS は国産で日本語サポートをアピールしやすい**。

### 3.2 資産の所在・スケーラビリティ

| 観点 | microCMS | Sanity | Keystatic |
|------|----------|--------|-----------|
| **資産の所在** | microCMS のクラウド。API でエクスポート可能 | Sanity のクラウド。エクスポート・API で取得可能 | **Git リポジトリ内**。 clone すればすべて手元に |
| **記事数・PV 増** | 記事数は実質無制限。**転送 20GB/月** がボトルネック。Team 移行で従量 | 10k doc・100GB まで無料。それ以上は有料プラン | **ファイル数・容量は Git/Vercel の範囲内**。ビルド時間増は ISR・ページネーションで緩和可能 |
| **コストの見通し** | Hobby 0円 → 転送増で Team 約4,900円/月〜 | Free のまま 500〜数千記事まで運用可能なことが多い | CMS 契約 0円。Vercel Pro のみ |

→ **将来のコスト・スケール**は **Sanity（無料枠の広さ）≒ Keystatic（契約なし）＞ microCMS（転送で有料化の可能性）**。

### 3.3 買い手の技術理解・継続運用

| 観点 | microCMS | Sanity | Keystatic |
|------|----------|--------|-----------|
| **技術スタックの説明** | 「Headless CMS（国産）。REST API で Next が取得」と説明可能 | 「Headless CMS（海外標準）。GROQ で取得」と説明しやすく、英語ドキュメントが豊富 | 「コンテンツはすべて Git のファイル。管理画面は Next に埋め込み」と説明可能 |
| **継続運用** | 買い手が microCMS の契約を引き継ぎ、そのまま運用 | 同様。Sanity の契約を引き継ぎ | 買い手がリポジトリを clone し、Vercel にデプロイすればそのまま運用。**追加契約なし** |

→ **Keystatic は「譲渡後の依存が最小」**。**Sanity は「Headless CMS」として一般的で、技術 DD で理解されやすい**。

### 3.4 将来見通しのまとめ

| 項目 | microCMS | Sanity | Keystatic |
|------|----------|--------|-----------|
| 譲渡の手続きの少なさ | ○ アカウント譲渡が必要 | ○ プロジェクト譲渡が必要 | ◎ リポジトリ譲渡のみ |
| 資産の明確さ | ○ クラウドにあるがエクスポート可能 | ○ 同様 | ◎ すべてリポジトリ内 |
| スケール時のコスト | △ 転送で有料化の可能性 | ◎ 無料枠が広い | ◎ CMS 契約なし |
| 買い手の理解のしやすさ | ◎ 国産・日本語 | ◎ 海外標準・英語ドキュメント | ○ Git ベースと説明すれば理解可能 |

## 5. 結論：どのケースでどれを選ぶか

### 優先軸ごとの推奨

| 最優先にしたいこと | 推奨 | 理由 |
|--------------------|------|------|
| **売却時の手続きを最小にしたい・資産をすべて Git に置きたい** | **Keystatic** | 譲渡＝リポジトリ譲渡のみ。外部 CMS 契約・アカウント移管が不要。管理の煩雑さも最小。 |
| **運用効率（日本語 UI・1本ずつ追加）を最優先したい** | **microCMS** | 国産・日本語で非エンジニアでも扱いやすい。一括投入も API で可能。転送量にだけ注意。 |
| **SEO・スケール・買い手の技術理解のバランスを取りたい** | **Sanity** | 無料枠が広く、API 一括投入が容易。海外標準で DD で説明しやすい。英語 UI は許容できる場合向け。 |

### 総合的な結論（1年後売却・自前コンテンツ投入を前提に）

- **「資産の所在を明確にし、売却時の説明と手続きを最小にしたい」**  
  → **Keystatic を第一候補**とするのが合理的。  
  運用は「1本ずつは /keystatic、大量はスクリプトでファイル生成＋Git」で回せる。反映が数分遅れる点だけ許容できれば、オペレーション・SEO・将来性のバランスが良い。

- **「まず運用の手軽さ（日本語・即時反映）を最優先し、売却はその次」**  
  → **microCMS** で開始し、転送量が増えたら Team へ移行する選択でよい。

- **「無料枠でたくさん記事を入れたい・API 一括投入を重視・買い手は技術に詳しい想定」**  
  → **Sanity** が適している。

**このプロジェクトの前提（事業売却の可能性・自前でコンテンツ投入・フクロウ・LINE は残す）を総合すると、「譲渡のシンプルさ」と「管理の煩雑さを増やさない」を重視するなら Keystatic、「運用の分かりやすさ」を最優先するなら microCMS、「スケールと API のバランス」を重視するなら Sanity** という整理になる。
