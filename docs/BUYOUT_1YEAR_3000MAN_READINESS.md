# 1年で3,000万円バイアウトに向けたサービス洗い出し・改善提案

**目標**: 1年以内に評価額3,000万円での事業売却（M&A）  
**買い手**: 不動産・リユース・終活・遺品整理業者など（「メディア＋ツール＋リスト」で即収益化したい企業）

---

## 1. 現状の整理（何ができているか・できていないか）

### 1.1  product（プロダクト）

| 項目 | 現状 | バイアウト観点 |
|------|------|----------------|
| **記事（SEO）** | 約18本（index.json）。戦略では Year1 150〜200本目標 | 記事数は目標の1割弱。流入の土台が薄い |
| **地域pSEO** | 地域×粗大ゴミ・補助金・相場の3種LP、全国市区町村、日本地図、生前整理コラム、パンくず・sitemap・robots 実装済み | ロングテールの受け皿は整っている。データ（areas.csv）の行数で規模が決まる |
| **診断ツール** | 実家じまい力・空き家リスク・相続準備力の3本。お手紙生成・LINE共有あり | PLG・バイラル要素はある。離脱後の再接触（メール等）は未整備 |
| **空き家シミュ** | 維持費目安・10年損失表示・査定CTA | 高単価リード（不動産）の入口として有効 |
| **資産・持ち物** | 登録・査定希望フラグ・写真査定CTA・お宝マップ（3件で共有） | 買取・査定送客のホットリスト化の土台 |
| **チェックリスト・エンディングノート** | localStorage 保存・進捗可視化・家族共有の入口 | 継続利用・登録の動機にはなるが、単体では収益に直結しない |
| **ガイドブック（PDF）** | 無料ダウンロードCTA・メール登録で「送った」扱い | リード獲得の軸。実際のPDF送付・メール配信の実装要確認 |

### 1.2 リード・データの永続化

| 項目 | 現状 | バイアウト観点 |
|------|------|----------------|
| **メール登録（/api/lead）** | POST で email, source 等を受付。Supabase が設定されていれば `leads` に保存。未設定時は console のみ | **本番で Supabase 未設定ならリードは一切残らない。** DD で「登録数」を示すには必須 |
| **リードイベント（/api/lead/event）** | 空き家シミュ・査定クリック・粗大ゴミクリック・資産高単価など。Supabase が設定されていれば `lead_events` に保存。未設定時は console のみ | S/A/B ランクの根拠データ。**未設定だとスコアリングも説明できない** |
| **anonymous_id ↔ email の紐付け** | 未実装。メール登録時に `anonymous_id` を leads や lead_events に書き込んでいない | 「この S ランクイベントは、のちにメール登録したユーザー」と説明できず、リストの質の説明が弱い |

### 1.3 収益・送客

| 項目 | 現状 | バイアウト観点 |
|------|------|----------------|
| **見積もり・査定の導線** | 「無料で見積もりを依頼する」「実家の資産価値を今すぐ無料で調べる」等はすべて **/guide（はじめかた）** への内部リンク | **外部の提携先（CPA・アフィ）への送客が無い。** 月間送客収益の数字が出せない |
| **提携先・CPA 契約** | コード上は「提携業者」「提携の専門家」等の文言のみ。実際のリンク・フォーム送信先は未実装 | 買い手は「送客単価・成約率」で評価する。**提携が無いと「ポテンシャル」のみの説明になる** |

### 1.4 可視化・運用

| 項目 | 現状 | バイアウト観点 |
|------|------|----------------|
| **リード・イベントのダッシュボード** | 未実装。Supabase を直接見るか、手動集計になる | **「月間リード数」「S/A/B 件数」「ソース別」を DD で示すには管理画面 or 定期レポートが欲しい** |
| **KPI の定義・ドキュメント** | SEO_CONTENT_STRATEGY に CVR・単価・目標値の記載あり。運用で更新されているかは不明 | 買い手向けに「何を KPI としているか」を1枚で説明できる資料があるとよい |

## 2. 3,000万円評価を得るために必要なこと（買い手の視点）

1. **トラフィックの規模と質**  
   月間 PV 数、オーガニック比率、直帰率。記事数・地域ページ数で「これから伸ばせる」ストーリー。

2. **リードの数と質**  
   メール登録数、イベント数（S/A/B）。**データが永続化されていないと「数」を証明できない。**

3. **収益の有無**  
   月間送客収益（CPA・紹介料）がゼロでも、「提携先とテスト導入済み」「送客フォーム・リンクは実装済み」であれば、買い手が即収益化しやすい。

4. **引き継ぎやすさ**  
   技術スタックが明確、リード・イベントのスキーマが明示、ダッシュボード or エクスポートで数字が見える。

## 3. 改善箇所（既存機能の強化）

| 優先度 | 改善内容 | 目的 |
|--------|----------|------|
| **高** | **リード・イベントの永続化を必須化** | Supabase 未設定時はリードが残らない。README またはデプロイ手順に「本番では Supabase 必須」と明記し、`leads` / `lead_events` の作成 SQL をリポジトリに含める。 |
| **高** | **メール登録時に anonymous_id を紐付ける** | フロントでメール登録成功時に `POST /api/lead` に `anonymous_id` を付与。API で `leads` に `anonymous_id` を保存。必要なら `lead_events` の同一 `anonymous_id` を「このリードの行動」として説明可能にする。 |
| **高** | **/guide を「送客のハブ」に変える** | 「無料で見積もりを依頼する」等の CTA を、提携先のフォーム URL または自前の申込フォーム（POST で提携先へ転送）に差し替え。まずは1社でも「送客リンク」を実装し、クリック数・送信数を計測できるようにする。 |
| **中** | **リード・イベントのエクスポート or 簡易ダッシュボード** | Supabase の `leads` / `lead_events` を CSV エクスポートする API、または認証付きの簡易管理画面。「月間リード数」「イベント種別別件数」を DD で提示できるようにする。 |
| **中** | **記事数の増強計画の可視化** | キーワード表・ブリーフテンプレを更新し、「あと何本で Year1 目標に届くか」をドキュメント化。週次で何本公開するかまで書いておくと、買い手に「実行力」が伝わる。 |
| **低** | **ガイドブック PDF の実際の送付** | メール登録後に PDF を添付またはダウンロードリンク付きメールを送る実装。未実装なら「登録ありがとう」メール＋PDF 配信の仕組みを用意する。 |

## 4. 新規実装の提案

| 優先度 | 新規実装 | 目的 |
|--------|----------|------|
| **高** | **送客申込フォーム（見積もり・査定）** | 自前で「見積もり希望」「不動産査定希望」等のフォームを設置。送信先はメール or 提携先 Webhook。送信数を「月間送客数」として計測し、買い手に示す。 |
| **高** | **提携先リンクの設定化** | 環境変数または config で「見積もり申込先 URL」「不動産査定申込先 URL」を管理。未契約でも「テスト用」URL を入れておき、クリック計測まで動かす。 |
| **中** | **リードスコア（S/A/B）の自動付与と一覧** | `lead_events` を元に、anonymous_id または email ごとに S/A/B を算出するバッチ or リアルタイムロジック。管理画面 or エクスポートで「Sランク○件」を一覧表示。 |
| **中** | **管理画面（認証付き）の最小版** | ログイン後に「今月のリード数」「今月のイベント数（種別別）」「S/A/B 件数」を表示。Supabase から取得。DD 時に「この画面で数字を見せられる」ようにする。 |
| **中** | **メール登録後のサンクスページ** | 登録完了後に遷移する専用ページで、「次のステップ（見積もり・査定・チェックリスト）」へのリンクを整理。コンバージョン経路を明確にする。 |
| **低** | **ニュースレター・再接触** | 登録メールに定期的なコンテンツやツールの案内を送る仕組み（SendGrid 等）。リードの「温め」とアクティブ率の向上。 |
| **低** | **AB テスト基盤** | CTA 文言や導線のバリエーションを出し分け、クリック率・送信率を比較。買い手に「改善余地」を示す。 |

## 5. 実装の優先順位（1年で3,000万を狙う場合）

1. **リード・イベントの永続化を本番で確実にする**（Supabase テーブル作成・env 設定・README 記載）  
2. **メール登録時に anonymous_id を送り、leads に保存する**  
3. **送客の「入口」を1つ作る**（見積もり or 査定のフォーム＋送信先。提携がなくても「テスト送信先」でよい）  
4. **リード・イベントの数字を見える化する**（CSV エクスポート API または簡易ダッシュボード）  
5. **記事を週3〜4本ペースで増やす**（キーワード表・ブリーフに沿った運用）  
6. **S/A/B ランクの自動付与と一覧**（管理画面 or エクスポートで「スコア付きリード」を提示）

## 6. ドキュメント・運用で揃えたいもの

- **README または DEPLOY 系ドキュメント**: 本番で Supabase を必須とし、`leads` / `lead_events` の作成手順を記載。テーブル定義 SQL は `docs/ARCHITECTURE_AND_ROADMAP.md`（leads）と `docs/PSEO_AND_PLG_ROADMAP.md`（lead_events）にあり。  
- **買い手向け1枚**: 「サービス概要」「主なKPI（リード数・イベント数・記事数・地域ページ数）」「技術スタック」「引き継ぎに必要な情報」を1ページにまとめた PDF または Markdown。  
- **KPI の定義書**: リード・送客・S/A/B の定義と、集計方法（どのテーブル・どのクエリで出すか）を短く文書化。  

---

このドキュメントをベースに、優先度の高い項目から実装・運用を進めると、1年後のバイアウト時に「数字と product の両方」を説明しやすくなります。
